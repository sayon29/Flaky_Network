System Architecture Summary: The Flaky Network

A breakdown of a real-time, low-latency UDP transport protocol.

1. High-Level Architecture

The system utilizes an asymmetric, real-time UDP streaming architecture designed to achieve the lowest possible playout delay while adhering to a strict 2.0x bandwidth cap. The design completely abandons TCP-like reactive retransmissions (ARQ). In a network where maximum jitter reaches 80ms, a reactive NACK (Negative Acknowledgment) incurs a severe Round-Trip Time (RTT) penalty, requiring a minimum of 40ms to 160ms just for transit. To achieve sub-90ms playout delays, the architecture allocates the entire 2.0x bandwidth budget to proactive Staggered Duplication, guaranteeing the redundancy is in flight before the receiver is even aware a drop has occurred.

  • The Sender (Event-Loop Multiplexer): Instead of blocking synchronously on recvfrom(), the sender operates an event loop driven by poll(). This decouples redundancy generation from the strict 20ms harness clock, allowing the sender to sleep for microsecond-precise intervals and inject arbitrary delays between primary and secondary packet transmissions.
  • The Receiver (Stateless Playout): The receiver operates with a highly lightweight seen array buffer. It eschews complex jitter buffers and reordering logic. Because redundancy is handled entirely on the wire, the receiver simply forwards the first valid arrival of any sequence number to the harness player; subsequent late or duplicated packets are instantly dropped.

2. The Evolution of the Design

To reach the theoretical physical limits of the network, the redundancy strategy underwent three major iterations to overcome specific network hazards.

Phase 1: Simple Piggybacking (N on N+1)

The initial design appended the payload of Frame N onto the packet for Frame N+1. While this incurred zero computational overhead, it mathematically locked the recovery time to the 20ms generation clock. If Packet N was lost, the backup physically could not leave the sender until t=20ms. Adding network transit time meant the minimum valid delay hovered around 40-50ms, which was too slow.

Phase 2: Offset Piggybacking (The Delay Trap)

To combat packet loss, the design initially pivoted to an offset model (e.g., placing Frame N inside Frame N+4 or N+5) under the assumption of burst losses. However, network profiling revealed the drops were actually independent events, not burst-correlated. Because this offset model mathematically locked the redundancy to the 20ms frame generation clock, the backup data was artificially delayed at the sender. Testing showed this design produced minimum playout delays of 120ms, rendering it uncompetitive, and it was quickly abandoned.

Phase 3: Decoupled Staggered Duplication

The final design abandoned piggybacking entirely. By moving to an event-loop (poll()) architecture, the sender was freed from the 20ms grid. The system now sends standard 164-byte UDP packets. The primary packet is sent at t=0, and a secondary "echo" packet is queued and fired at a precise millisecond offset (e.g., t=1ms). This establishes a tight "race condition," allowing the backup to dodge instantaneous network hiccups while arriving well before a strict 80ms deadline.

The Bandwidth Budget Hack

The 160-byte payload is cryptographically hashed, rendering it perfectly incompressible. The bandwidth cap is strictly 2.0x (320 bytes per frame).

Raw Packet: 164 bytes (4-byte seq + 160-byte payload)
100% Duplication: 164 * 2 = 328 bytes (2.05x overhead - INVALID)

Because 328 bytes exceeds the 320-byte budget, the sender cannot mathematically afford to duplicate every frame. The architecture solves this using modulo arithmetic, skipping the staggered echo on exactly 1 out of every 20 frames.

(19 frames * 328 bytes) + (1 frame * 164 bytes) = 6396 bytes per 20 frames
6396 / 20 = 319.8 bytes average per frame.
319.8 / 160 = 1.99x Overhead (VALID)

The Joint Probability Calculation (The 81ms Floor)

Profile B exhibits a maximum network delay/jitter of 80ms and a baseline drop rate of roughly 5%. The system achieves an 81ms delay by exploiting independent joint probabilities. To fail an 81ms deadline, both the primary packet and the staggered echo (sent at t=1ms) must individually fail (either dropped, or delayed beyond 80ms).

  • Primary Packet (Deadline 81ms): Fails if dropped (5%) or delayed by 81ms (impossible under Profile B). Failure rate = ~5%.
  • Echo Packet (Deadline 80ms due to 1ms stagger): Fails if dropped (5%) or delayed by 80ms (a 1-in-60 chance based on uniform 20-80ms jitter). Failure rate = ~6.6%.

Because these are independent network routing events, the probability of both packets failing is their product: 0.05 * 0.066 = 0.0033 (0.33%). Even factoring in the 5% of frames that lack an echo (due to the bandwidth budget), the overall systemic deadline-miss rate resolves to ~0.93%, safely passing the 1.0% cap.

4. System Breaking Points

Because the architecture is highly optimized to ride the absolute limits of the network's physical constraints, it is vulnerable to two specific boundary conditions:

  1. Jitter Physics (The +1ms Failure): The 81ms score for Profile B is calculated exactly as 80ms (Max Network Jitter) + 1ms (Echo Stagger Offset). If the relay applies an 81ms delay spike to the network, the staggered echo will arrive at 82ms. Therefore, increasing the network's maximum delay by even 1 millisecond mathematically breaks this design, pushing the deadline misses past 1%.
  2. The 10% Packet Loss Threshold: This architecture relies on squaring the packet loss probability. If the network's baseline packet loss increases to 10%, the failure rate of the staggered block becomes 0.10 * 0.10 = 0.01 (1.0%). A loss rate of 15% results in a 2.25% failure rate, rendering the system INVALID. Surviving >10% loss requires abandoning simple duplication for block-based XOR Erasure Coding across 3 or more packets, which would force the minimum playout delay back over 100ms.
